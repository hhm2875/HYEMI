## 🤾‍♂️ 트러블슈팅
문제1. 학교 네트워크 AP Isolation으로 인한 통신 단절
문제점: 설계상 라즈베리파이 → Node.js(HTTP POST) → MySQL 구조였으나, 학교 네트워크의 AP Isolation 정책으로 동일 네트워크 내 기기 간 통신이 차단되어 Pi에서 서버로 센서 데이터 전송이 불가능했음.
원인 분석: 네트워크 계층 문제임을 확인하기 위해 Pi와 서버 간 ping, 포트 스캔 등으로 통신 가능 여부를 점검하고, AP 설정상 클라이언트 간 트래픽이 차단되는 구조임을 확인.
해결: 단기적으로 Pi → MySQL 직접 INSERT 방식으로 우회하여 데이터 수집 파이프라인을 유지했고, 이후 네트워크 환경 재구성 후 원래 설계인 Pi → Node.js API → MySQL 구조로 복귀. 이 과정에서 두 가지 아키텍처 모두를 검증해본 경험을 확보함.
문제2. 가스센서(MQ-2) 노이즈로 인한 반복적인 오탐
문제점 설명: MQ-2 센서가 2초 주기로 ±4~11 단위의 자연 진동을 보이는데, 초기 사용하던 임계값(p90 기준 약 11)이 이 노이즈 범위와 거의 겹쳐 정상 상태에서도 위험 알림이 반복적으로 발생함.
해결방안: SQL 윈도우 함수로 3개 샘플 이동평균(gas_avg3, temp_avg3)을 계산하고, 히스테리시스 로직(GAS_GAP, TEMP_GAP)과 연속 상승 카운터를 도입해 노이즈로 인한 순간적 변동은 무시하고 지속적인 상승 패턴만 위험으로 판단하도록 개선함.
문제3. 판단 주체 이원화로 인한 대시보드 표시 불일치
문제점 설명: 라즈베리파이의 룰베이스 판단(고정 임계값)과 서버의 AI 적응형 판단(aiJudge.js, 데이터 기반 임계값)이 서로 다른 기준을 사용하면서, 같은 센서값을 두고 두 시스템이 다른 위험도를 판단하는 모순이 발생함.
해결방안: 역할을 분리하여 Pi는 센서 데이터 수집·전송만 담당하고, 최종 위험도 판단은 서버의 AI 적응형 모델로 일원화함으로써 판단 기준 불일치 문제를 해소함.
문제4. IR 불꽃센서 실제 유효거리와 스펙 간 차이
문제점 설명: 스펙시트상 유효거리보다 실제 테스트 결과 5~10cm 이내에서만 반응하는 것을 확인, 플로팅 GPIO 핀으로 인한 오탐도 함께 발견됨.
해결방안: 직접 실측을 통해 실제 반응 거리를 파악하고 센서 배치 및 테스트 방식을 이에 맞게 조정함. 스펙시트에만 의존하지 않고 실측 검증을 거쳐 신뢰도를 확보함.
문제5. Git 공유 브랜치 강제 푸시 사고
문제점 설명: 변수명 일괄 변경(smoke→gas) 작업 중 실수로 팀 공유 master 브랜치에 강제 반영되어 다른 팀원들의 작업 내역과 충돌 발생.
해결방안: git reset --hard + --force-with-lease로 이전 상태 복구, 팀원들은 git fetch origin 후 git reset --hard origin/master로 재동기화하여 문제 해결.
